## 1.0 INTRODUCTION

### 1.1 Purpose

The purpose of this document is to define the business requirements for enhancing the existing SAP 1099 MISC extract program for Company Code 0001. The current system extracts all withholding payments, including non-reportable Virtual Card payments, and lacks optimal formatting for names, addresses, and emails. By implementing targeted exclusion rules and improving the reporting layout, the Corporate Tax team can ensure IRS filings are accurate, compliant, and easy to review, ultimately saving time during year-end reconciliation.

---

## 2.0 PROJECT OVERVIEW

### 2.1 Objectives

- **Specific:** Exclude Virtual Card payments, separate recipient name fields, prioritize P.O. Box addresses, include vendor emails, and offer ALV Grid Excel extraction.
- **Measurable:** Eliminate 100% of Virtual Card transaction lines from the output and provide 100% properly structured name outputs.
- **Attainable:** Leverages existing tables in SAP without requiring new custom Z-tables.
- **Realistic:** Resolves critical reporting blockers using existing functionality.
- **Time-bound:** Required before Q4 reporting begins.

### 2.2 Assumptions

- Testing is performed in QA prior to P41 production transport.
- Business rules primarily apply to Company Code 0001 (Primary variant).

### 2.3 Constraints

- Other forms such as 1024S, 1099-NEC, or 1099-B are out of scope.
- Changes must seamlessly integrate into the existing background processes without performance degradation.

### 2.4 Risks

- If address details or email addresses are not properly maintained in the Vendor Master, the reporting output might contain missing fields. Coordinated data cleansing with the Master Data team may be required.

---

## 3.0 SUPPORTING DOCUMENTS

### 3.1 As Is

Currently, when the year-end 1099 MISC process is executed:

1. All vendor payments with withholding tax codes are extracted into a flat file.
2. Virtual Card payments (Payment Method = 'B'), which are non-reportable via 1099s, are incorrectly included in the output summary.
3. Names and addresses are pulled generically without consideration for vendor type (Individual vs. Company) or preferred mailing address (P.O. Box vs. Physical).
4. Vendor emails are completely absent from the extract, making follow-up communication manual and tedious.
5. Users must parse raw flat files instead of reviewing structured Excel spreadsheets.

### 3.2 To Be

Once the enhancement is deployed:

1. The report selection screen will allow users to dynamically exclude Virtual Card payments.
2. Individual vendor names will be split structurally, while Company names will correctly utilize Name 1 / Name 2 fields.
3. The report will prioritize P.O. Box addresses to reduce returned mail and will output vendor emails natively.
4. The Corporate Tax team will be able to view and export the data directly into a structured ALV Grid Excel file.

## 4.0 REQUIREMENTS

### 4.1 Business Rules/Requirements

| BRD ID | Business Rule/Requirement |
| :--- | :--- |
| BR-001 | Users must be able to exclude Virtual Card (Payment Method = 'B') transactions from the 1099 extract. |
| BR-002 | The report must correctly split the recipient name based on the vendor type (Individual 'S' vs. Company 'T'). |
| BR-003 | The report must display the P.O. Box address first, and only use the physical street address if the P.O. Box is blank. |
| BR-004 | The vendor email address must be included in the final data output. |
| BR-005 | The report must be exportable to Microsoft Excel via a standard ALV grid. |

### 4.2 Stakeholder Requirements

- **Corporate Tax:** Requires clean, compliant 1099 data that correctly identifies names and addresses while ignoring non-reportable Virtual Card payments.
- **Corporate Accounting:** Needs the tool to be exportable into Excel to facilitate easier tracking and reconciliation against vendor payment history.

### 4.3 Solution Requirements

- The enhancement must adjust the data selection logic within the 1099 Program to apply the stated filters and logic rules.
